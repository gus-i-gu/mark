## 2. Evidence reconciliation

Implemented locally at the inspected head:

- `local-account` is a provisional local owner placeholder, not an authenticated Account;
- a fresh database creates and persists an app-private UUID v4 Device identity;
- the Device UUID is generated by the app and is not a hardware identifier;
- Purchase registration atomically writes Purchase facts, one immutable `SyncEvent`, and one `PendingEvent` in state `pending`;
- Device sequence is monotonic for that persisted Device and is distinct from event ID and Purchase occurrence time;
- `SyncState.accountCursor` exists but is initially empty;
- local copy says normal use needs no network, synchronization is inactive, and local use sends no user data or usage analytics;
- deterministic CSV/PDF export exists elsewhere in the product and is user-initiated.

Not implemented or validated: authenticated Account ownership, enrollment/revocation, upload/download, `SubmissionId`, server acceptance, acknowledgement, inbox application, cursor replay, retry processing, duplicate suppression at a server, conflict resolution, convergence, relay retention/deletion, account deletion or two-Device operation. Enum names `uploading`, `accepted`, and `failed` are structural vocabulary only; no current behavior earns those user-visible claims.

The earlier Cycle 10 local-consolidation checkpoint describes synchronization as a non-goal. The newer human/Main opening charter supersedes that scope for investigation only. It does not supersede the truthful implementation boundary.

## 3. Vocabulary map

| Concept | Required meaning | Must not imply |
| --- | --- | --- |
| Local fact | Authoritative fact committed in this installation's local database | already copied elsewhere |
| Synchronized copy | A fact accepted through the protocol and applied to another authorized Device | backup completeness or immediate equality |
| Local database | Offline-first operational store used by this installation | server cache or shared database |
| Server coordination state | Bounded relay/protocol records used to accept and deliver updates | the user's only history or developer analytics store |
| Account | Authenticated ownership/security boundary, model undecided | `local-account`, email, Person, or one Device |
| Installation | One installed app/data-store context | a person or hardware identity |
| Device | App-private enrolled protocol participant associated with an installation | hardware serial, advertising ID, or nickname |
| Device UUID | Opaque app-generated protocol identity | hardware fingerprint or secret credential |
| Event ID | Stable identity of one immutable synchronization fact | retry attempt, sequence position, or cursor |
| SubmissionId | Candidate stable identity of one logical submission/retry series | accepted event identity; exact relation unresolved |
| Device sequence | Monotonic order of events produced by one Device | global order, occurrence time, or server cursor |
| Cursor | Opaque server-issued download position | Device sequence or proof every Device applied an event |
| Pending | Safely stored locally and waiting for an upload attempt | network transfer in progress |
| Uploading | A transport attempt is currently in progress | server acceptance |
| Accepted | Server validated, authorized and durably recorded the event | another Device downloaded or applied it |
| Acknowledged | A named receiving Device confirmed application through a defined cursor/ack rule | universal acknowledgement unless policy proves it |
| Applied | Downloaded fact committed to this local database | acknowledged by all Devices |
| Duplicate | The same logical event/submission was seen again and caused no second effect | conflict or validation failure |
| Retry | Another attempt using the same stable retry identity and content | creation of a replacement event |
| Conflict | Individually valid facts cannot both be applied under the accepted entity policy | malformed input or temporary network failure |
| Validation failure | Request/content violates a contract before acceptance | concurrent edit conflict |
| Offline | Device is intentionally or observably without usable connectivity; local work continues | server outage or signed-out state |
| Unreachable | A connection to the service could not be completed | invalid credentials or rejected content |
| Authentication failure | Credentials are absent, expired or rejected | offline state |
| Synchronization | Authorized exchange, deduplication and deterministic application of facts | backup, export, or merely having a queue |
| Backup | Recoverable independent copy governed by restore guarantees | synchronization or relay retention |
| Export | User-initiated readable artifact such as CSV/PDF | backup/restore or automatic upload |

## 4. User-visible state matrix

These are semantic requirements for minimum non-deceptive functional feedback, not a page or visual design.

| Internal situation | Preferred visible state | Explanation and safe action |
| --- | --- | --- |
| local operation, no network | `Saved on this device` | `You can continue offline. This update has not been copied to another Device yet.` |
| queued | `Waiting to upload` | `The update is safe on this device. Markei will try when the service is reachable.` Offer `Try now` only when implemented. |
| active attempt | `Uploading…` | `Keep Markei open until this attempt finishes; your local copy remains saved.` Cancellation consequences require a contract. |
| server accepted | `Uploaded` | `The server accepted this update. Another Device may not have applied it yet.` Never label this `Synchronized everywhere`. |
| accepted, no peer acknowledgement | `Uploaded · waiting for another Device` | Name the relevant Device only if its safe user-visible label is defined. Explain that offline Devices can be behind. |
| downloaded and committed locally | `Downloaded and applied` | `This device now includes the update.` Do not imply all derived views are fresh unless rebuilt successfully. |
| repeated delivery | `Already applied` | `Markei recognized the same update and did not add it twice.` No warning styling unless user action is needed. |
| resolvable input rejection | `Could not upload this update` | State the invalid field/rule, preserve local facts, and offer correction before retry. |
| domain conflict | `Needs attention` | Identify the affected fact and available non-destructive choices. Never silently overwrite or call it a duplicate. |
| service unreachable | `Service unavailable` | `Your changes remain on this device. Check the connection and try again later.` |
| credentials expired | `Sign in again to continue updating Devices` | Preserve local operation; do not claim data loss. Keep secret/diagnostic detail out of copy. |
| Device revoked | `This Device can no longer update the Account` | Explain whether local viewing/export remains possible and route to re-enrollment or Account owner action. |
| interrupted, outcome unknown | `Upload outcome unknown` | `The update is safe locally. Check synchronization status before retrying; Markei must reuse the same submission identity.` |
| retention expired/gap | `Updates are missing from the server` | `This Device cannot safely catch up from the retained updates.` Offer only an accepted rebuild/re-enrollment/restore route. |
| retry succeeds idempotently | `Update confirmed` | Explain that the repeated attempt did not create a second fact when useful; avoid technical jargon by default. |

State changes must never discard local user input. Every failure message follows:

```text
[operation/state] + [known plain-language reason] + [what remains safe]
+ [one useful next action] + [diagnostic reference when support needs it]
```

Raw exceptions, tokens, connection strings, payloads, account UUIDs and Device UUIDs do not belong in ordinary feedback. A diagnostic reference may identify a log entry without revealing credentials.

## 5. Privacy and retention language

Truthful local baseline now: `Markei stores your data locally on this device. Normal local use does not require a network. Synchronization is not active in this build.` Current `no user data or usage analytics` wording is valid only for the local implementation and must be revised before any networked build is presented.

Candidate networked wording, contingent on implementation and policy evidence:

`When you turn on inter-device updating, Markei sends the required Product and Purchase updates through the Markei service so your authorized Devices can receive them. The service keeps coordination data only for the stated retention period. Markei does not use this information for development-side usage analytics.`

Required separation:

- server retention means protocol data held for delivery/recovery under a stated duration and deletion rule;
- developer analytics means separate collection/use for measuring behavior; absence of analytics does not mean absence of synchronization data;
- deletion from relay storage does not delete local Purchase history;
- deletion from one Device does not imply deletion from the Account or other Devices;
- synchronization is not backup, and relay retention must not be described as disaster recovery;
- export remains explicit and user-initiated; synchronization must not be called export;
- rebuildable Lists projections should be recomputed from synchronized Product/Purchase facts, not treated as authoritative synchronized inventory.

Before release, evidence must establish payload scope, encryption in transit, authentication/authorization, retention start/end, acknowledgement basis, expiry behavior, account deletion, logs/backups, provider access, and recovery after an unrecoverable gap. Exact retention time and deletion guarantees remain unresolved Main/Design decisions.

## 6. Conflict and derived-data semantics

Product and Purchase facts require entity-specific rules. An immutable accepted Purchase event is not corrected by a blanket last-write-wins label. Duplicate delivery repeats the same identity/content; conflict involves competing facts or policy. Validation failure is rejection before acceptance. Main/Design must decide correction and convergence behavior before user copy can name choices.

Lists (`Storage`, `Shortage`/`Ending soon`, `Market`/`Expected ended`, `Not enough history`) are rebuildable personal projections from compatible registered history. Synchronize the facts needed to rebuild them, then report projection refresh separately. `Lists updated` must not mean measured inventory, and a projection calculation failure must not roll back an already applied Purchase fact.

## 7. MCG-01 through MCG-04 learning prerequisites

### MCG-01 — isolated Neon environment

Developer must understand project/branch/database/role separation, least privilege, server-only credentials and environment isolation. Easily confused: Neon branch versus Git branch; database role versus Markei Account; connection string versus safe evidence. Never commit, paste, screenshot or log credentials. Evidence: provider resource names, role scopes, migration result and a redacted/fingerprinted connectivity probe. Keep authentication, authorization, managed PostgreSQL and secret handling Yellow/Red until direct learner explanation and safe transfer are observed.

### MCG-02 — API runtime

Developer must understand client/API/database trust boundaries, TLS endpoint, runtime secret injection, migration identity and disposal. Easily confused: public API URL versus private database URL; build-time value versus runtime secret; connectivity versus authorization. Evidence: variable names and redacted presence, server-origin database probe, denied client/direct access test, and reproducible teardown—never secret values or raw headers.

### MCG-03 — Account and Device test identities

Developer must understand Account authentication, installation-to-Device enrollment, credential lifecycle, revocation and historical Device identity. Easily confused: email/Person/nickname with Account; Device UUID with credential or hardware ID; test token with durable secret. Evidence: two distinct redacted identities, enrollment/revocation outcomes and scoped authorization probes without token disclosure. Keep identity, authentication and authorization maturity unchanged absent learner evidence.

### MCG-04 — two-Device evidence environment

Developer must understand fixture isolation, local01/local02 ownership, event/submission/sequence/cursor distinctions, acknowledgement, retention observation, cleanup and unknown outcomes. Easily confused: server acceptance with peer application; duplicate with retry; sync with backup. Evidence: named redacted fixtures, event/state timeline, controlled interruption/duplicate/gap results, local database comparisons and teardown record. Do not expose Purchases, credentials or provider secrets in screenshots/logs.

At every MCG, stop before external mutation, provide a minimal human checklist, retain the last runnable repository checkpoint, and continue only after explicit confirmation. Successful configuration is project evidence, not learner maturity.

## 9. Misconceptions to prevent

- `pending` does not mean uploading; `uploaded` does not mean applied elsewhere;
- local `accepted` enum presence is not server acceptance evidence;
- Device UUID is neither hardware identity nor authentication credential;
- event ID, SubmissionId, Device sequence and cursor are not interchangeable;
- retry with the same identity is not a duplicate business fact;
- authentication failure is not offline; validation failure is not conflict;
- synchronization is neither backup nor export;
- bounded relay retention is not `no server data` and not developer analytics;
- relay deletion does not erase local history;
- Lists projections are rebuildable outputs, not synchronized inventory facts;
- passing sync tests would be project evidence, not learner mastery.
