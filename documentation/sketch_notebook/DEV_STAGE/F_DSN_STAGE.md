# F_DSN_STAGE — Correlated Client/API Diagnostic Boundary

> Unit: C10-MCG02-TRANSPORT-OBSERVABILITY_20260721
> Authority: Main Chat
> Status: READY FOR MATERIALIZATION

## 1. Architecture objective

Instrument the existing port/adaptor path without coupling Closure widgets to HTTP, Fastify or
Drift rows.

Preferred dependency direction:

```text
Closure UI
  -> Closure runner/application command
       -> hosted connection-check port
       -> authentication/token port for protected operations
       -> Sync coordinator
       -> diagnostic-attempt repository

HTTP adapters
  -> validated origin and route construction
  -> bounded transport observation

Fastify boundary
  -> sanitized request lifecycle observer
  -> existing authorization/application/database services
```

Codex may refine names after source inspection, but must preserve this direction.

## 2. Correlation design

Use one per-invocation correlation value generated by the client and sent through the existing
`x-correlation-id` mechanism for health and protected operations where the route contract permits.
The full value may exist transiently in transport memory but must not be persisted, displayed or
logged. Derive one deterministic short fingerprint suitable for matching Closure to Render logs.

The derivation must:

- be stable for one invocation across client display and server logs;
- not reveal or permit reconstruction of Account, Device, Event or Submission identity;
- accept only bounded input and sanitize control characters;
- resist log injection and accidental collision well enough for the small diagnostic window;
- be tested with malformed and adversarial input.

Do not reuse SubmissionId, EventId, DeviceId, AccountId or a request hash as the correlation value.

## 3. Transport observation boundary

Create or refine a typed transport observation/result rather than leaking implementation
exceptions into application/UI code. It must separate:

- request not started;
- request started, no response headers;
- response headers/status received;
- body received but protocol invalid;
- parsed stable hosted response;
- cancellation;
- timeout and supported IO/TLS/DNS/connect failures;
- unexpected local/programming failure.

Do not claim that the server did not apply a request merely because the client timed out.

Inspect the existing enrollment and Sync transports for duplicated origin validation, path joining,
deadline and error translation. Extract a shared policy only if this reduces divergence without
expanding the unit into a transport rewrite. Otherwise document the exact comparison and make the
minimum bounded correction.

## 4. Connection-check boundary

Implement the health probe through a dedicated port/adaptor, not through SyncTransport and not by
constructing HTTP inside the widget. It calls the existing public health routes, carries no access
token, and returns typed liveness/readiness observations.

The adapter must use the same configured hosted origin validation as the real hosted transports so
its success is relevant. A successful check must not touch Sync coordinator/outbox or authenticated
route state.

## 5. Durable diagnostics

The attempt repository owns persistence of bounded diagnostic state. If additive columns or a new
table are needed, migrate forward without rewriting the historical opaque attempt. Reopen must
preserve exactly one historical attempt and all new terminal attempts.

Application orchestration must guarantee:

- one confirmed invocation -> one attempt;
- stage progression is monotonic;
- terminal finalization happens once even on exceptions;
- cancellation before confirmation creates no attempt;
- UI refresh reads persistence and causes no network activity;
- queue/submission transitions remain owned by the existing Sync repositories.

## 6. Server logging boundary

Do not simply set unrestricted `logger: true`. Add a repository-owned sanitized lifecycle logger or
strict Fastify redaction/serialization policy with tests. Logs must observe request ingress and
completion without serializing raw request objects.

Preserve route inventory and authorization ordering. For protected routes, logging may state that
authentication was accepted/rejected but must not observe or serialize credentials. Database-stage
events belong at the application/repository boundary only when the transaction actually starts.

Logging failure must not change the application response or transaction semantics.

## 7. Invariants

Preserve:

- exact unknown-submission retry identity and outbox state;
- immutable Event/Device sequences and content;
- strict JWT/JWKS and enrollment checks;
- RLS/runtime least privilege;
- existing public health-route exception and protected route inventory;
- Closure/Settings separation;
- offline-first behavior;
- no provider-specific secrets in source.

## 8. Design evidence required in I

Report:

- final dependency graph and changed ports/adapters;
- exact URL/path construction comparison for enrollment, Sync and health;
- correlation generation/fingerprinting and privacy boundary;
- transport failure taxonomy and unknown-outcome boundary;
- durable schema/migration decision;
- Fastify hook/application logging placement and redaction policy;
- timeout decisions;
- all preserved invariants and residual risks.
